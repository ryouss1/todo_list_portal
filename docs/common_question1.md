### 1.2 既存共通システムの設計パターン

| パターン | 内容 |

## 2. 設計方針の質問

### Q1: RBAC（ロールベースアクセス制御）のスコープ

既存システムは3層の権限モデルを持つ:
1. **ユーザー → ロール** (t_user_role): 複数ロール付与可
2. **ロール → メニュー** (t_role_menu): ロール単位の権限
3. **ユーザー → メニュー** (t_user_menu): ユーザー個別オーバーライド

**質問:** portal_core ではどこまで実装するか？

- **案A: 全3層** — 既存システムと同等の柔軟性（複雑度: 高）
- **案B: 2層のみ** — ユーザー→ロール + ロール→メニュー。ユーザー個別オーバーライド(t_user_menu)は省略（複雑度: 中）
- **案C: 1ロール + パーミッション** — spec_roadmap の Phase A 案（roles + role_permissions + user_roles）で、メニューDB管理なし（複雑度: 低〜中）

→ **個人的には案Bが現実的と考える。** 案Aのユーザー個別オーバーライドは運用が複雑で、実際にはほとんど使わないケースが多い。

### Q2: system_id（マルチシステム識別）

既存システムでは `m_role.system_id` と `m_menu.system_id` でシステム単位にロール・メニューをスコープしている。

**質問:** portal_core に `system_id` 概念を入れるか？

- **案A: 入れる** — roles/menus テーブルに system_id カラム追加。将来の倉庫システム等で同一DBを共有可能
- **案B: 入れない** — アプリごとに独立したDB/インスタンスを前提。portal_core 自体はシステム区別しない

→ 倉庫システムと Todo List Portal が同じ DB を共有する想定はあるか？ それとも別インスタンス/別DB？

### Q3: メニューのDB管理

既存システムの m_menu は DB でメニュー定義を管理（URL、表示名、親子関係、表示件数、適用日）。
portal_core の NavItem は Python コードで登録（DB に永続化しない）。

**質問:** メニューを DB 管理に移行するか？

- **案A: 完全DB管理** — メニューマスタテーブルを作成し、NavItem を廃止。管理画面でメニュー編集可能（複雑度: 高）
- **案B: ハイブリッド** — NavItem（コード登録）は残し、権限チェックだけ DB で管理。メニューの表示/非表示を role_menus テーブルで制御（複雑度: 中）
- **案C: コード維持** — NavItem に `required_permission` を追加するだけ。DB テーブル追加なし（複雑度: 低）

→ **案Bを推奨。** メニュー定義自体はコードで管理するのが開発効率的に良く、権限だけ DB で管理すれば柔軟性とシンプルさのバランスが取れる。

### Q4: kino_kbn（機能区分）の粒度

既存システムでは `t_role_menu.kino_kbn` と `t_user_menu.kino_kbn` で、メニューごとの操作レベルを制御している（推定: 0=参照, 1=登録/更新, 2=削除等）。

**質問:** portal_core の権限粒度はどうするか？

- **案A: メニュー単位のみ** — あるロールがあるメニュー(ページ)にアクセスできるかどうかだけ（閲覧可/不可の2値）
- **案B: アクション単位** — メニューごとに read / write / delete 等のアクションを定義
- **案C: リソース+アクション** — `require_permission("users", "delete")` のような汎用モデル（spec_roadmap Phase A 案）

→ 既存システムの `kino_kbn` は実際にどういう値が入っているか？ 0〜2 の3段階？ それとも別のパターン？

### Q5: 部署（組織）構造

既存システムの m_busho は事業部→センター→課→係の4階層。portal_core の groups は名前のみのフラット構造。

**質問:** 組織構造をどこまで拡張するか？

- **案A: 4階層固定** — 既存システムと同じ jigyobu/center/ka/kakari 構造
- **案B: 再帰的（ツリー）** — `parent_id` による任意深度のツリー構造
- **案C: groups 拡張** — 現在の groups に `parent_id` と `level` を追加する最小拡張
- **案D: 維持** — groups はそのまま。組織管理が必要なアプリは独自テーブルで管理

→ 倉庫システムでの組織構造はどの程度の階層が必要か？ 拠点→倉庫→エリア のような構造？

### Q6: 社員マスタの扱い

既存システムでは m_user（認証用）と m_shain（人事用）が分離されている。社員には会社コード、入退社日、職種区分、日英カナ名前等がある。

**質問:** portal_core に社員概念を入れるか？

- **案A: 入れる** — employees テーブルを追加。users ↔ employees を紐付け（1:1 or 1:0）
- **案B: users を拡張** — users テーブルに employee_code, company_code 等のカラムを追加
- **案C: 入れない** — 社員管理が必要なアプリは独自テーブルで管理

→ 倉庫システムで社員コード・会社コードは必要か？ 入退社日の管理は portal_core で行うか？

### Q7: 適用日管理（有効期間パターン）

既存システムはほぼ全マスタに `tekiyo_kaishi_date` / `tekiyo_shuryo_date` がある。これにより「○月○日からこのロールが有効」「この部署は○月末で廃止」等の時間軸管理が可能。

**質問:** portal_core に適用日パターンを導入するか？

- **案A: 主要テーブルに導入** — roles, user_roles, departments 等に有効期間を追加
- **案B: 不要** — 現在の `is_active` / 物理削除で十分

→ 実運用で「○月から新ロールが有効」のような時間軸管理は必要か？

### Q8: 設定マスタ（m_settei 相当）

既存システムの m_settei は DB ベースの key-value 設定。言語別、階層的（親設定コード）で、属性型や桁数指定もある。

**質問:** portal_core にランタイム設定テーブルを追加するか？

- **案A: 汎用設定テーブル** — m_settei 相当の key-value テーブル。管理画面で設定変更可能
- **案B: シンプル版** — `settings(key, value, description)` のフラットテーブル
- **案C: 不要** — Python の CoreConfig / .env で十分

→ ランタイムで変更したい設定にはどのようなものがあるか？

### Q9: お知らせ機能（t_oshirase 相当）

既存システムにはシステム単位のお知らせ機能がある（タイトル、本文、適用期間、言語、臨時区分）。

**質問:** portal_core にお知らせ機能を追加するか？

- **案A: 追加** — ダッシュボードやログイン画面にお知らせを表示
- **案B: 不要** — 既存のアラート機能で代替

→ 「メンテナンス予告」「システム更新のお知らせ」等を管理者が投稿する機能が必要か？

### Q10: パスワード履歴（t_pass 相当）

既存システムの t_pass はパスワード変更履歴を保持し、過去N回の再利用を防止できる。
portal_core は password_reset_tokens があるがパスワード履歴は管理していない。

**質問:** パスワード再利用防止を実装するか？

- **案A: 実装** — password_history テーブルで過去 N 回のハッシュを保持、変更時に照合
- **案B: 不要** — 現在のパスワードポリシー（長さ・複雑度）で十分

### Q11: 監査証跡の強化

既存システムは全テーブルに `ins_user`, `ins_term`（端末名）, `upd_user`, `upd_term` がある。
portal_core は `created_at` / `updated_at` のみ（誰が・どこから、の情報なし）。

**質問:** 監査カラムを強化するか？

- **案A: 全テーブルに追加** — `created_by`, `updated_by` (FK to users), `created_from`, `updated_from` (IP/端末)
- **案B: 重要テーブルのみ** — roles, user_roles 等のセキュリティ関連テーブルだけ
- **案C: 不要** — 既存の auth_audit_logs で認証操作のログは取れている

**解答** 案A

### Q12: ICカード・営業日カレンダー

既存システムには ICカード管理（m_card / t_card）と 営業日カレンダー（m_calendar）がある。

**質問:** これらは portal_core に含めるか？

- ICカード: 勤怠の `input_type=ic_card` と連携可能だが、portal_core の範疇か？
- 営業日カレンダー: 既存の calendar_events（イベント管理）とは概念が異なる。営業日/休日の判定に使う？

## 3. 優先度と実装順序の提案

上記を踏まえた実装優先度の案:

| 優先度 | 機能 | 対応する質問 | 理由 |
|--------|------|------------|------|
| **P1** | RBAC（ロール + 権限） | Q1, Q4 | 倉庫システムで最も必要な基盤 |
| **P1** | ナビ権限フィルタ | Q3 | RBAC と連動、画面アクセス制御 |
| **P2** | 組織構造の拡張 | Q5 | 部署・拠点管理 |
| **P2** | お知らせ機能 | Q9 | 比較的シンプルで汎用的 |
| **P3** | パスワード履歴 | Q10 | セキュリティ強化 |
| **P3** | 設定マスタ | Q8 | ランタイム設定の需要次第 |
| **P4** | 社員マスタ | Q6 | 倉庫システムの要件次第 |
| **P4** | 適用日管理 | Q7 | 運用が複雑になるため慎重に検討 |
| **P4** | 監査証跡強化 | Q11 | あると良いが全テーブル改修は大規模 |
| **P5** | system_id | Q2 | マルチアプリ要件が確定してから |
| **P5** | ICカード・営業日 | Q12 | アプリ固有の可能性が高い |

## 4. 回答待ち事項のまとめ

1. **Q1** — RBAC は3層/2層/1層のどれ？
    **解答** 案A
2. **Q2** — 倉庫システムと Todo List Portal は同一DB共有？ 別インスタンス？
    **解答** 別DB
3. **Q3** — メニュー定義は DB 管理 / ハイブリッド / コード維持？
    **解答** 案A
4. **Q4** — `kino_kbn` の実際の値は？ 操作レベルの粒度は？
    **解答** kino_kbnは 0,1 しかない 操作レベルは 案C
5. **Q5** — 組織の階層深度は？ 倉庫システムの組織構造は？
    **解答** 案B
6. **Q6** — 社員マスタは portal_core に入れるか？
    **解答** 必要
7. **Q7** — 適用日管理は必要か？
    **解答** 社員管理のため必要
8. **Q8** — ランタイム設定の具体的な要件は？
    **解答** 案A 管理画面なし
9. **Q9** — お知らせ機能は必要か？
    **解答** いまはしない
10. **Q10** — パスワード再利用防止は必要か？
    **解答** いまはしない
11. **Q11** — 監査証跡の強化レベルは？
    **解答** 案A
12. **Q12** — ICカード・営業日カレンダーの要否は？
    **解答** ICカードはやらない、営業日は、カレンダーに適用したいのでやる
