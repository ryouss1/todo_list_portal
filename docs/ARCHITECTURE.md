## 1. 現状の課題

現在のシステムは「シングルユーザーのプロトタイプ」として機能しているが、マルチユーザー・マルチ機能のシステムに拡張するには以下の根本的な課題がある。

### 1.1 課題一覧

| # | カテゴリ | 課題 | 深刻度 |
|---|---------|------|--------|
| 1 | ~~認証~~ | ~~認証機構が存在しない（`DEFAULT_USER_ID` 固定）~~ | ~~高~~ **解決済み** |
| 2 | 認可 | データアクセス制御がない（誰でも全データ参照可能） | 高 |
| 3 | ~~ユーザーコンテキスト~~ | ~~全ルーターが `DEFAULT_USER_ID` をハードコードしている~~ | ~~高~~ **解決済み** |
| 4 | ~~サービス層~~ | ~~ビジネスロジックがルーター層とCRUD層に散在~~ | ~~中~~ **解決済み** |
| 5 | WebSocket | 単一用途（ログ配信のみ）、認証なし、ユーザー区別なし | 中 |
| 6 | バックグラウンド処理 | 非同期タスク・定期実行の仕組みがない | 中 |
| 7 | フロントエンド | Vanilla JSで重複コード多い、状態管理なし | 中 |
| 8 | ~~DBマイグレーション~~ | ~~マイグレーションツール未導入（手動ALTER TABLE）~~ | ~~中~~ **解決済み** |
| 9 | エラーハンドリング | フロントエンドのエラー処理が最小限 | 低 |
| 10 | 監視 | ヘルスチェック・メトリクス収集の仕組みがない | 低 |

### 1.2 現状アーキテクチャの強み（活かすべき点）

- FastAPIの依存性注入（`Depends`）パターンが既に確立されている
- CRUD層の一貫したパターン（拡張しやすい）
- Pydanticスキーマによる型安全なバリデーション
- WebSocketの基盤が既にある（拡張で対応可能）
- テストの基盤が既にある（conftest.pyのDB分離パターン）

## 2. 検討すべき主要な設計判断

### 2.1 認証方式の選定

マルチユーザー対応の最優先課題。全機能の前提となる。

| 方式 | 概要 | メリット | デメリット |
|------|------|---------|-----------|
| **セッション + Cookie** | サーバー側でセッションを保持 | 実装がシンプル、ブラウザ標準のCookie管理 | サーバー側に状態を持つ、スケールアウト時に共有ストアが必要 |
| **JWT** | トークンをクライアント側で保持 | ステートレス、APIとの親和性が高い | トークン失効の即時反映が困難、ペイロードサイズ |
| **セッション + Cookie（推奨）** | - | - | - |

**推奨: セッション + Cookie方式**

理由:
- 本システムはブラウザベースのWebアプリケーションであり、外部API公開の予定がない
- サーバーサイドレンダリング（Jinja2テンプレート）との相性が良い
- セッションの即時無効化が可能（ログアウト、権限変更時）
- WebSocketの認証もCookieで自然に対応できる
- 現時点で水平スケールアウトの必要性が低い（必要になった場合はRedisセッションストアで対応可能）

**実装方針:**

```
[ブラウザ] → Cookie(session_id) → [FastAPI] → session_store → [ユーザー情報]
```

- パスワードハッシュ: bcrypt（`passlib`ライブラリ）
- セッション管理: サーバーサイドセッション（初期は辞書 or DB、将来的にRedis）
- 認証ミドルウェア: FastAPIの `Depends` で `get_current_user` を全エンドポイントに注入

### 2.2 認可モデルの設計

ユーザーの権限に基づくアクセス制御。

**ロールベースアクセス制御（RBAC）を採用:**

| ロール | 権限 |
|--------|------|
| `user` | 自分のデータのCRUD、公開Todoの閲覧、他ユーザーの日報閲覧、在籍状態の閲覧 |
| `admin` | 上記に加え、全ユーザーの勤怠閲覧、ユーザー管理、アラートルール管理 |

**データアクセス制御の方針:**

| データ種別 | 閲覧 | 操作 |
|-----------|------|------|
| プライベートTodo | 本人のみ | 本人のみ |
| 公開Todo | 全ユーザー | 作成者のみ |
| 勤怠記録 | 本人 + admin | 本人のみ |
| 在籍状態 | 全ユーザー | 本人のみ |
| 日報 | 全ユーザー | 作成者のみ |
| タスク | 本人のみ | 本人のみ |
| ログ | 全ユーザー | システム（API経由） |
| アラート | 全ユーザー | admin（ルール管理） |

### 2.3 WebSocketアーキテクチャ

現在はログ配信のみだが、在籍記録・アラートにもリアルタイム性が必要になる。

**現在の課題:**
- WebSocketManagerが1つのグローバルインスタンスで全クライアントにブロードキャスト
- ユーザー区別なし、チャネル概念なし

**拡張方針: チャネルベースのWebSocket**

```
[クライアント] ──→ WS /ws/{channel} ──→ [WebSocketRouter]
                                              ├── channel: "logs"      → ログのリアルタイム配信
                                              ├── channel: "alerts"    → アラートの即時通知
                                              └── channel: "presence"  → 在籍状態の変更通知
```

| チャネル | 配信対象 | 配信タイミング |
|---------|---------|--------------|
| `logs` | 全接続クライアント | ログ作成時 |
| `alerts` | 全接続クライアント | アラート発生・更新時 |
| `presence` | 全接続クライアント | 在籍状態変更時 |

実装案:
- `WebSocketManager` をチャネル対応に拡張（チャネルごとの接続リスト管理）
- 接続時にCookieから認証情報を取得
- 単一のWebSocketエンドポイント `/ws` でメッセージ内のtypeフィールドで購読チャネルを切り替える方式、または `/ws/{channel}` でチャネルごとにエンドポイントを分ける方式

### 2.4 バックグラウンド処理

ログファイル収集やアラート判定にはバックグラウンド処理が必要。

| 候補 | 概要 | メリット | デメリット |
|------|------|---------|-----------|
| FastAPI BackgroundTasks | リクエスト処理後にバックグラウンド実行 | 追加ライブラリ不要、シンプル | リクエスト起点のみ、定期実行不可、永続化なし |
| APScheduler | Python製のスケジューラ | 軽量、FastAPIとの統合が容易 | 単一プロセス、分散実行不可 |
| Celery + Redis/RabbitMQ | 分散タスクキュー | 高機能、スケーラブル、定期実行対応 | インフラ追加が必要（Redis等） |

**推奨: 段階的な導入**

- **Phase 1（初期）**: FastAPI BackgroundTasks + APScheduler
  - ログファイルの定期収集: APSchedulerで定期実行
  - アラート判定: ログ受信時にBackgroundTasksで評価
  - 追加インフラ不要で開始できる

- **Phase 2（規模拡大時）**: Celery + Redis に移行
  - ログの大量取り込み、アラートの複雑な条件評価
  - 分散処理が必要になった場合

### 2.5 データベースマイグレーション

テーブル追加・変更が多数発生するため、マイグレーション管理は必須。

**推奨: Alembic の導入**

- SQLAlchemy公式のマイグレーションツール
- 既存のSQLAlchemyモデルとの親和性が高い
- マイグレーションの自動生成が可能
- バージョン管理によるロールバック対応

導入タイミング: ログイン機能の実装開始前（最初のスキーマ変更前）

### 2.6 フロントエンドの方針

| 選択肢 | メリット | デメリット |
|--------|---------|-----------|
| **現状維持（Vanilla JS + Jinja2）** | 学習コスト不要、追加ツール不要 | コード重複増加、状態管理困難、開発効率低下 |
| **SPAフレームワーク導入（React / Vue）** | コンポーネント再利用、状態管理、開発効率 | 学習コスト、ビルドツール導入、API設計の変更 |
| **Jinja2 + htmx（推奨）** | サーバーサイドレンダリング維持、少量のJS、学習コスト低 | 複雑なインタラクションには限界あり |

**推奨: 現状の Vanilla JS + Jinja2 を維持しつつ段階的に改善**

理由:
- 現在のシステム規模（画面数7→14程度）ではSPA化のコストが見合わない
- ただし、JS共通関数の統合（`escapeHtml` 等の重複排除）は即座に実施すべき
- 画面数がさらに増加する場合にSPA化を再検討

改善ポイント:
- `static/js/common.js` に共通関数を集約
- APIクライアントの統一（認証ヘッダの一括管理）
- エラーハンドリングの共通化

## 3. 目標アーキテクチャ

### 3.1 全体構成図

```
                          [ブラウザ]
                         /    |    \
                   HTTP  /    |WS   \ HTTP
                       /     |      \
                [ページ] [リアルタイム] [API]
                      \      |      /
                       \     |     /
                    [FastAPI Application]
                     |       |      |
              [認証MW]  [ルーター層]  [WSルーター]
                     |       |      |
                  [サービス層（ビジネスロジック）]
                     |       |      |
                  [CRUD層（データアクセス）]
                     |       |      |
                [PostgreSQL] [Redis]  [ファイルシステム]
                                        (ログファイル)
                     |
              [バックグラウンド処理]
              (APScheduler / Celery)
```

### 3.2 レイヤー構成の変更

**現在:**
```
ルーター → CRUD → モデル → DB
```

**目標:**
```
ルーター → サービス → CRUD → モデル → DB
             ↓
       認証・認可チェック
```

サービス層を追加することで:
- ビジネスロジック（タイマー計算、出勤ルール判定等）をルーターから分離
- 認可チェック（自分のデータかどうか等）を一元管理
- 複数のCRUD操作をまたぐトランザクション管理
- WebSocket通知のトリガーを集約

### 3.3 ディレクトリ構成（目標）

```
app/
├── auth/                          # 認証・認可（新規）
│   ├── dependencies.py            #   get_current_user, require_admin 等
│   ├── session.py                 #   セッション管理
│   └── password.py                #   パスワードハッシュ
├── services/                      # サービス層（拡張）
│   ├── websocket_manager.py       #   WebSocket管理（チャネル対応に拡張）
│   ├── todo_service.py            #   Todo ビジネスロジック

│   ├── task_service.py            #   タスク ビジネスロジック
│   ├── report_service.py          #   日報 ビジネスロジック（新規）
│   ├── alert_service.py           #   アラート ビジネスロジック（新規）
│   └── log_collector.py           #   ログファイル収集（新規）
├── models/                        # モデル層（テーブル追加）
│   ├── ...（既存）
│   ├── presence.py                #   在籍状態（新規）
│   ├── daily_report.py            #   日報（新規）
```

## 5. 技術選定まとめ

### 5.1 追加ライブラリ（予定）

| ライブラリ | 用途 | 導入フェーズ |
|-----------|------|------------|
| `alembic` | DBマイグレーション管理 | Phase 0 |
| `passlib[bcrypt]` | パスワードハッシュ | Phase 1 |
| `itsdangerous` or `python-jose` | セッショントークン署名 | Phase 1 |
| `apscheduler` | 定期実行スケジューラ | Phase 4 |
| `redis`（任意） | セッションストア・WebSocket PubSub（スケール時） | 必要時 |

### 5.2 技術判断サマリー

| 判断項目 | 決定 | 理由 |
|---------|------|------|
| 認証方式 | セッション + Cookie | ブラウザアプリとの親和性、WebSocket認証の容易さ |
| 認可モデル | RBAC（user / admin） | シンプルかつ必要十分 |
| フロントエンド | 現状維持（Vanilla JS + Jinja2） | 規模に対してSPA化コストが見合わない |
| WebSocket | チャネルベースに拡張 | ログ・アラート・在籍の3系統を統一管理 |
| バックグラウンド処理 | APScheduler（初期）→ Celery（必要時） | 段階的に複雑さを導入 |
| マイグレーション | Alembic | SQLAlchemy標準、既存パターンとの親和性 |
| セッションストア | メモリ/DB（初期）→ Redis（必要時） | 初期は追加インフラ不要で開始 |

## 6. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| 認証導入時の全ルーター改修 | 既存機能のリグレッション | テストカバレッジを先に拡充、段階的に適用 |
| スキーマ変更による既存データ破損 | データロスト | Alembic導入を最優先、変更前にバックアップ |
| WebSocket拡張時の複雑化 | デバッグ困難 | チャネルごとのユニットテスト、接続状態のログ強化 |
| バックグラウンド処理の信頼性 | ログ取り込み漏れ、アラート未配信 | 処理結果のDB記録、リトライ機構、死活監視 |
| フロントエンドのJS肥大化 | 保守性低下 | 共通化を先行実施、一定規模超過時にSPA化を判断 |
| 同時編集の競合 | データ不整合 | 楽観的ロック（updated_at比較）の導入を検討 |
