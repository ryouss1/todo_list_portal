# ログファイル収集機能 — 質問事項 (Q&A)

> 本ドキュメントは [spec_log_function.md](./spec_log_function.md) の補足資料です。

## Q1. ログの保持期間・削除ポリシー

**現状**: `logs` テーブルに DELETE/ARCHIVE API が存在せず、データが無限に蓄積される。

**質問**:
1. ログの保持期間を設定すべきか？（例: 30日、90日、無期限）
**解答** 90日(設定ファイルで変更できるように)
2. 自動削除（TTL ベース）を実装すべきか？それとも手動アーカイブ機能か？
**解答** 自動削除
3. 削除は物理削除でよいか？監査上、一定期間の保持義務はあるか？
**解答** 物理削除
4. 削除・アーカイブの権限は admin のみでよいか？
**解答** YES

**影響**: テーブル肥大化によるクエリ性能劣化、ストレージコスト

## Q2. ログ POST API の認証とレート制限

**現状**: `POST /api/logs/` は認証不要。外部システムからのログ投入を容易にするための設計だが、悪意のあるリクエストに対して無防備。

**質問**:
1. API キーベースの認証を導入すべきか？（例: `X-API-Key` ヘッダー）
**解答**:基本 POSTは最初は実装しない、認証を用意(APIキー)
2. レート制限を設けるべきか？（例: IP あたり 100 req/min）
**解答**:なし

3. 投入元の IP アドレスホワイトリストは必要か？
**解答**:なし

4. 現状の認証不要運用で問題ないと判断してよいか？
**解答**:APIKEY

**影響**: セキュリティ、DoS 耐性、不正ログ投入防止

## Q5. ログソースのファイルパス一意制約

**現状**: 複数のログソースが同一の `file_path` を指すことが可能。同じファイルから重複したログが取り込まれるリスクがある。

**質問**:
1. `file_path` に UNIQUE 制約を追加すべきか？
**解答**:同じファイル取込問題の対策をするけどUNIQUE制約じゃない。
DBではファイルのタイムスタンプを管理、タイムスタンプをDBに保存する。
DBに全件取込を行い、行数ベースで、管理を行う。1回目取得時10行目まで取得したら2回目は11行目から取得
新規ファイルがあれば更に取込を行う。

データ構造ノイメージ(これをベースに提案して)
LOG管理テーブル
ログ管理ID,サーバ名、アクセス方法(FTP,WIN),ID,PASS,フォルダ名、アクセス間隔(60sec-300sec)

LOGヘッダテーブル
ログ管理ID,ファイル名,タイムスタンプ

LOG明細
ログ管理ID,行数,ログ

 
3. 同一ファイルを異なる parser_pattern で2回パースするユースケースはあるか？（例: アクセスログとエラーログが混在するファイル）
**解答**:ない

4. UNIQUE 制約を追加する場合、DB マイグレーションの影響は許容可能か？
**解答**:しない

**影響**: データ重複防止、運用の明確化

追加仕様として2パターンのログ収集方法を検討
1: フォルダのタイムスタンプとフィルタ条件のみ管理しログファイルの中身はみない。
　(ログをDBに保存するのはUSERが参照したタイミングのみ)
2: 特定のファイル名パターンのみ、ログを全件、DBに登録する(Errファイル等)

これにより膨大なログをDBに保存する必要がなく、必要時に任意に取得にする事ができる。
DBのログから色々な調査機能は作れないがログ取得データの最小化が見込める。

これにより、WEBのログ表示画面のUIも変更となる。
ログルール定義毎に、アクセスタイムスタンプのみを表示するのみにすることによって
膨大なログルールを閲覧性を高くすることができる。

例) SYSTEM名 ログフォルダ名 更新日付 のみ画面に表示
上の例よりもっと閲覧性を高くしたいので、良い案を提案して

補足情報
　ログの取得方法は以下でいく予定。
  1 サーバから取りに行くパターン(FTP,winのみ)
  2 APIに各システムが送信するパターン（こちらは当初は実装しない)

この要件を元に再度設計しなおして。
