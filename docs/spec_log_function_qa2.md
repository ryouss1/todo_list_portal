## Q1. ページネーション方式の選定

**背景**: Q&A v1 の Q3 で「オフセットベースとカーソルベースのどちらが適切か判断する資料を作って」と回答いただいた。

### 比較表

| 項目 | オフセットベース (`limit` + `offset`) | カーソルベース (`before`/`after` + `limit`) |
|------|--------------------------------------|-------------------------------------------|
| **実装難易度** | 低（SQL の `LIMIT/OFFSET`） | 中（WHERE 条件 + ORDER BY） |
| **UI との相性** | ページ番号指定に最適 | 無限スクロールに最適 |
| **性能（少量データ）** | 良好 | 良好 |
| **性能（大量データ）** | offset 値が大きいと遅い（全行スキャン） | 一定速度（インデックス利用） |
| **途中挿入の影響** | ページズレが発生する | 影響なし |
| **任意ページへのジャンプ** | 可能（`?offset=500`） | 不可（前後移動のみ） |
| **total_count 取得** | 容易（`COUNT(*)`） | 別途クエリが必要 |

### 本システムでの考慮事項

- `log_entries` は全件取込のログのみ → データ量は制御可能（90 日自動削除）
- ファイルビューアは行番号ベース → オフセットが自然（`?offset=500&limit=500` = 501 行目から）
- 検索 API は管理系機能 → 超高速レスポンスは不要
- 「ページ 3 を見る」「500 行目から見る」といった UI が想定される

**提案**: **オフセットベース**を採用（実装が単純、UI と相性が良い、データ量は 90 日制限で制御）

**質問**:
1. オフセットベース（提案）でよいか？
**解答** カーソル
2. カーソルベースを採用する理由はあるか？
**解答** ログの閲覧性向上のため

## Q2. ログ検索条件の詳細

**背景**: Q&A v1 の Q3 で「検索・フィルタ機能の追加は必要、検索条件は要検討」と回答いただいた。

### 提案する検索条件

| 条件 | パラメータ名 | 型 | 説明 |
|------|-------------|-----|------|
| ソース指定 | `source_id` | integer | ログソース ID フィルタ |
| 重要度 | `severity` | string | severity 完全一致（例: ERROR） |
| 重要度グループ | `severity_min` | string | 指定以上の severity（例: WARNING → WARNING + ERROR + CRITICAL） |
| キーワード | `keyword` | string | メッセージの部分一致（LIKE `%keyword%`） |
| 取込日時（開始） | `from_date` | datetime | received_at の開始日時 |
| 取込日時（終了） | `to_date` | datetime | received_at の終了日時 |
| ファイル指定 | `file_id` | integer | 特定ファイルのエントリのみ |

### 検索対象の範囲

- **full_import** で DB に保存された `log_entries` のみが検索対象
- **metadata_only** のファイル内容は検索対象外（DB にデータがないため）
- 全文検索（PostgreSQL の `tsvector`/`tsquery`）は初期実装では見送り

**質問**:
1. 上記の検索条件で十分か？追加の条件はあるか？
**解答** TYPE ( WEB, HT , BATCH , OTHER)
2. `severity_min` （指定以上のseverity一括フィルタ）は必要か？それとも `severity` 完全一致のみでよいか？
**解答** 重要度グループなし、
3. メッセージ検索は部分一致（LIKE）でよいか？正規表現検索は必要か？
**解答** LIKE
4. 検索結果の表示形式は、ファイルビューア内のハイライト表示か、独立した検索結果一覧か？
**解答** ハイライト

## Q3. ファイルアクセス制限方式

**背景**: Q&A v1 の Q8 で「LOG_ALLOWED_PATHS がどういう変数か説明して」「拡張子制限等」と回答いただいた。

### LOG_ALLOWED_PATHS の説明（v1 の設計）

v1 では、ローカルファイルシステムのファイルパスを直接監視する設計だったため、`LOG_ALLOWED_PATHS` は「サーバ上のどのディレクトリのファイルを監視対象として登録できるか」を制限する環境変数でした。

**例**:
```
LOG_ALLOWED_PATHS=/var/log/,/opt/app/logs/
```
→ `/var/log/httpd/error.log` は登録可能、`/etc/passwd` は登録不可

### v2 での変更点

v2 ではリモートサーバへの FTP/SMB 接続に変更されたため、ローカルパスの制限は不要になりました。代わりに以下の制限方式を提案します:

| 制限方式 | 説明 | 例 |
|---------|------|-----|
| **拡張子ホワイトリスト** | 対象ファイルの拡張子を制限 | `.log`, `.txt`, `.csv`, `.err` |
| **拡張子ブラックリスト** | 特定の拡張子を除外 | `.exe`, `.dll`, `.bin`, `.zip` |
| **ファイルサイズ上限** | 個別ファイルのサイズ制限 | 100 MB（設定可能） |
| **file_pattern** | ログソースごとの glob パターン | `*.log`, `error*.txt` |

**質問**:
1. 拡張子制限はホワイトリスト方式（許可する拡張子のみ）とブラックリスト方式（禁止する拡張子のみ）のどちらが適切か？
**解答** ホファイトリスト
2. デフォルトで許可/禁止する拡張子の一覧はあるか？
**解答** .logのみ許可
3. ファイルサイズの上限値はいくらが適切か？（提案: 100 MB）
**解答** 100MB
4. `file_pattern`（glob パターン）のみで十分か？拡張子制限は別途必要か？
**解答** 十分

## Q4. FTP 接続の詳細設定

**背景**: v2 で FTP 接続を新規追加。運用環境に合わせた設定が必要。

### 確認事項

| 項目 | 選択肢 | 提案 |
|------|--------|------|
| モード | アクティブ / パッシブ | パッシブ（NAT/FW 環境で安全） |
| 暗号化 | 平文 FTP / FTPS (TLS) / SFTP | 平文 FTP（初期実装、将来 FTPS 対応） |
| タイムアウト | 接続/読取タイムアウト | 接続: 30 秒、読取: 60 秒 |
| 文字コード | パス名のエンコーディング | UTF-8（FTP サーバ側の設定に依存） |

**質問**:
1. FTP のモードはパッシブモード固定でよいか？アクティブモードが必要な環境はあるか？
**解答** パッシブ固定
2. FTPS (FTP over TLS) は初期実装で必要か？それとも将来対応でよいか？
**解答** 不要
3. SFTP (SSH File Transfer Protocol) のサポートは必要か？（`paramiko` ライブラリが必要）
**解答** 不要
4. 接続タイムアウト・読取タイムアウトの値は適切か？
**解答** 適切

## Q5. SMB 接続の詳細設定

**背景**: v2 で Windows ファイル共有 (SMB) 接続を新規追加。

### 確認事項

| 項目 | 選択肢 | 提案 |
|------|--------|------|
| パス形式 | UNC 形式 (`\\server\share\path`) / POSIX 形式 | 登録時はフォルダパスのみ（`logs$\sql\`） |
| 認証 | NTLM / Kerberos | NTLM（初期実装） |
| SMB バージョン | SMB2 / SMB3 | SMB3（`smbprotocol` デフォルト） |
| ドメイン | 認証にドメイン名が必要か | オプショナル |

**質問**:
1. SMB のパス入力形式はどうすべきか？（例: `logs$\sql\error` のようなフォルダパスで、ホスト名は別フィールド `host` に入力）
**解答** POSIX方式

2. ドメイン認証は必要か？（Active Directory 環境の場合）必要な場合、`domain` カラムを追加
**解答** 任意

3. Kerberos 認証のサポートは必要か？
**解答** 不要

4. SMB 接続で対象となるサーバの OS バージョンは？（Windows Server 2012 以降？）
**解答** Windows 2003以降

## Q6. 接続エラー時のリトライ方針

**背景**: リモートサーバへの接続はネットワーク障害、サーバダウン等で失敗する可能性がある。

### 提案

| 項目 | 値 |
|------|-----|
| リトライ回数 | 3 回（設定可能） |
| リトライ間隔 | 指数バックオフ（5 秒 → 10 秒 → 20 秒） |
| エラー通知 | 連続 N 回失敗でアラート生成 |
| エラー記録 | `log_sources.last_error` に最新エラーを記録 |

**質問**:
1. リトライ回数は 3 回で適切か？
**解答** YES

2. 連続エラー時にアラートを自動生成すべきか？生成する場合、何回連続失敗でアラートにするか？
**解答** 3回

3. 接続エラーが続く場合、自動的にソースを無効化（`is_enabled=false`）すべきか？
**解答** YES

## Q7. オンデマンド参照のキャッシュ・性能

**背景**: `metadata_only` モードではユーザー操作時にリモートサーバに接続してファイル内容を読取る。複数ユーザーが同時に同じファイルを閲覧する場合の性能が懸念される。

### 想定シナリオ

1. ユーザー A がファイルを開く → FTP/SMB 接続 → 500 行取得（2-5 秒）
2. ユーザー B が同じファイルを開く → 再度 FTP/SMB 接続 → 500 行取得（2-5 秒）
3. ユーザー A がページ送り → 再度接続 → 次の 500 行取得（2-5 秒）

### 提案するキャッシュ方式

| 方式 | 説明 | メリット | デメリット |
|------|------|---------|-----------|
| **キャッシュなし** | 毎回リモートから取得 | 常に最新、実装が単純 | 遅い、サーバ負荷 |
| **メモリキャッシュ** | TTL 付きで一定時間保持 | 高速、サーバ負荷軽減 | メモリ消費、データ鮮度 |
| **一時ファイルキャッシュ** | ローカルに一時ファイルとして保存 | 大容量対応 | ディスク消費、管理複雑 |

**質問**:
1. キャッシュなし（毎回リモートから取得）で運用可能か？
2. キャッシュが必要な場合、TTL（有効期間）はどの程度が適切か？（例: 5 分）
3. 同時接続ユーザー数の想定は？（少人数 5 名以下なら問題なし）
4. リモートファイルの読取に 2-5 秒のレイテンシーは許容可能か？

上の案ではなく、タイムスタンプのみ管理の場合は、ログファイルは各自USERがアクセスして取得しサーバでログを管理しない。
タイムスタンプのみ管理　（サーバログなし、USERにはパスのみ提供、USERは各々そのパスにアクセスしてファイルを取得)
ファイルも管理（定期的にファイルをDBに保存し、DBで管理、DBでは行数で管理を行う）WEB上でもログが参照できる。
そのため、WEBブラウザ上から任意でログを取得する機能はなしとする。

## Q8. 削除済みファイル (status=deleted) の処理

**背景**: 設計書 セクション 7.2 で「status=deleted かつ log_entries が 0 件の log_files レコードは削除候補（要検討）」と記載。

### 提案

| 方式 | 説明 |
|------|------|
| A: 即座に削除 | リモートに存在しないファイルの log_files レコードを次回スキャンで即座に物理削除 |
| B: 保持期間後に削除 | status=deleted から N 日後に物理削除（ファイルが復活する可能性を考慮） |
| C: 削除しない | status=deleted のまま永続保持（履歴として残す） |
**解答** : C

**質問**:
1. status=deleted のファイルレコードはどう処理すべきか？（A/B/C）
2. B の場合、保持期間は何日が適切か？（提案: 30 日）
3. 削除されたファイルが同名で再作成された場合、新規ファイルとして扱うべきか？
**解答** サーバのファイルは一切編集しない、読むだけ

## Q11. アラート非同期キューの実装方式

**背景**: Q&A v1 の Q9 で「非同期評価にすべき、ルール数 100 件を想定」と回答いただいた。

### 提案する実装方式

| 方式 | 説明 | メリット | デメリット |
|------|------|---------|-----------|
| **A: asyncio.Queue** | インメモリキュー（Python 標準） | 実装が単純、追加依存なし | プロセス再起動で消失 |
| **B: デバウンス方式** | N 秒ごとにバッチ評価 | 効率的、DB 負荷軽減 | 遅延が発生 |
| **C: PostgreSQL LISTEN/NOTIFY** | DB のイベント通知 | 永続化、復旧可能 | 実装が複雑 |

**提案**: 方式 A（asyncio.Queue）を採用。ログ取込時にキューに追加し、バックグラウンドタスクがキューからデキューして評価。再起動時のキュー消失は、再起動前後のログで評価漏れが発生するが、運用上許容可能と判断。

**質問**:
1. asyncio.Queue（インメモリ）でよいか？再起動時の評価漏れは許容可能か？
**解答** A

2. キューの最大サイズは設定すべきか？（提案: 10,000 件、超過時は古いものから破棄）
**解答** 10000
3. 評価遅延はどの程度まで許容可能か？（提案: 最大 5 秒）
**解答** 5秒

## Q12. テスト方針

| テスト種別 | 対象 | 方式 |

| 優先度 | テスト内容 |

| 中 | ファイル一覧・コンテンツ取得 API |
| 中 | スキャナーのメタデータ差分更新ロジック |
| 低 | FTP/SMB 接続の結合テスト |

**質問**:
1. FTP/SMB の接続テストはモックのみでよいか？実サーバでの結合テストも必要か？
**解答** YES
2. テストカバレッジの目標はあるか？
**解答** なし
3. テスト用の FTP/SMB サーバ環境はあるか？
**解答** 後で提示
